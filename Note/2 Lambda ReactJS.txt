To deploy a **Next.js application on AWS Lambda using Docker**, the best production-ready path is to containerize the app and run it as a **Lambda Container Image**. This works well for SSR/API routes and avoids many zip-package limits.

---

# ✅ Architecture

**Next.js App** → **Docker Image** → **Amazon ECR** → **AWS Lambda** → **API Gateway / Function URL**

---

# ✅ Step 1: Create Next.js App
    npx create-next-app nextjs-lambda-app
    cd nextjs-lambda-app

# ✅ Step 2: Install Server Adapter
    Use a lightweight server like `serverless-http` with Express, or run standalone mode.

    For Next.js 13+/12+, easiest is standalone output.

    Update `next.config.js`

    module.exports = {
        output: "standalone",
    };
```

---

# ✅ Step 3: Build App

```bash
npm run build
```

This creates:

```bash
.next/standalone
.next/static
public/
```

---

# ✅ Step 4: Create Lambda Server File

Create `server.js`

```js
const { createServer } = require("http");
const next = require("next");

const app = next({
  dev: false,
  conf: { distDir: ".next" }
});

const handle = app.getRequestHandler();

let server;

exports.handler = async (event, context) => {
  if (!server) {
    await app.prepare();
    server = createServer((req, res) => {
      handle(req, res);
    });
  }

  return {
    statusCode: 200,
    body: "Next.js running on Lambda"
  };
};
```

# ✅ Better Approach (Recommended)

Use **OpenNext** (best for Lambda + Next.js)

```bash

```bash
npx open-next build
```

---

# ✅ Step 5: Dockerfile for Lambda

Create `Dockerfile`

```dockerfile
FROM public.ecr.aws/lambda/nodejs:18

WORKDIR ${LAMBDA_TASK_ROOT}

COPY package*.json ./
RUN npm install --production

COPY . .

RUN npm run build

CMD ["server.handler"]
```

---

# ✅ Step 6: Build Docker Image

```bash
docker build -t nextjs-lambda .
```

---

# ✅ Step 7: Test Locally

```bash
docker run -p 9000:8080 nextjs-lambda
```

Invoke:

```bash
curl http://localhost:9000/2015-03-31/functions/function/invocations -d '{}'
```

---

# ✅ Step 8: Push to Amazon ECR

```bash
aws ecr create-repository --repository-name nextjs-lambda
```

Login:

```bash
aws ecr get-login-password --region ap-south-1 | docker login --username AWS --password-stdin <account-id>.dkr.ecr.ap-south-1.amazonaws.com
```

Tag:

```bash
docker tag nextjs-lambda:latest <account-id>.dkr.ecr.ap-south-1.amazonaws.com/nextjs-lambda:latest
```

Push:

```bash
docker push <account-id>.dkr.ecr.ap-south-1.amazonaws.com/nextjs-lambda:latest
```

---

# ✅ Step 9: Create Lambda from Container

```bash
aws lambda create-function \
--function-name nextjs-lambda \
--package-type Image \
--code ImageUri=<ECR_IMAGE_URI> \
--role arn:aws:iam::<account-id>:role/lambda-role
```

---

# ✅ Step 10: Add API Gateway / Function URL

Simplest:

```bash
aws lambda create-function-url-config \
--function-name nextjs-lambda \
--auth-type NONE
```

`open-next` helps because **raw Next.js is not designed to run directly on AWS Lambda efficiently**. Next.js expects either:

    * a persistent Node.js server (`next start`)
    * Vercel’s platform runtime
    * static hosting for exported sites

    AWS Lambda is **ephemeral, stateless, request-driven**, and has packaging/runtime constraints. `open-next` adapts Next.js for that environment.

---

# ✅ What OpenNext Does

## 1. Converts Next.js build into AWS-friendly artifacts

When you run:

```bash
npx open-next build
```

It transforms your Next.js app into deployable components like:

* Lambda handlers for SSR pages / App Router rendering
* Separate assets for static files
* Image optimization function
* ISR revalidation support
* CloudFront/S3 compatible output

So instead of trying to run `next start`, you run AWS-native components.
