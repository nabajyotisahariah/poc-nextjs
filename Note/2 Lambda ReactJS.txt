Deploying a Next.js app on AWS Lambda is typically done using a serverless adapter (since Next.js isn’t natively designed for Lambda). The most efficient and production-grade approach is via Serverless Framework + serverless-next.js or newer solutions like OpenNext.

1. Choose Deployment Strategy

   Recommended options:
   1. OpenNext (modern, actively maintained)
   2. serverless-next.js (older but widely used)

  👉 Prefer OpenNext for new projects.

2. Prepare Next.js App

   Ensure your app is production-ready:
	npm run build

   Key checks:
    	Avoid unsupported Node APIs

	Use next.config.js properly:
	module.exports = {
  	     output: "standalone"
	};

3. Install Required Dependencies
	npm install open-next

4. Create OpenNext Config
    Create:
	open-next.config.ts

    Example:
	import { defineConfig } from "open-next";
	
	export default defineConfig({
  	   aws: {
    		region: "ap-south-1",
  	   },
	});

5. Build for Lambda
   npx open-next build

   This generates:	
	Lambda handlers
	CloudFront config
	Optimized assets

6. Deploy to AWS

   Option A: Using OpenNext CLI
	npx open-next deploy

   Option B: Using Serverless Framework
	npm install serverless

	Create serverless.yml:
	service: nextjs-lambda-app

	provider:
  		name: aws
  		runtime: nodejs20.x
  		region: ap-south-1

	functions:
  	    nextApp:
    		handler: .open-next/server-function/index.handler
    		events:
      		  - httpApi: '*'

	package:
  	   patterns:
    		- '!node_modules/**'
    		- '!.next/cache/**'
    		- '!**/*.map'
    		- '!**/*.log'

	
	Deploy
		serverless deploy

7. Configure CloudFront (Critical)

    Lambda alone is not enough.

    You need:
	CloudFront (CDN)
	S3 for static assets
	Lambda@Edge or regional Lambda

	OpenNext usually provisions:

	CloudFront distribution
	Edge functions
	Static asset routing

8. Environment Variables

    For stages (dev/prod):
	.env.production
	.env.development

    With Serverless:

	provider:
  	    environment:
    		NODE_ENV: production

9. Optimize for Lambda
   Key optimizations:
   Use ISR (Incremental Static Regeneration)
   Prefer static pages (getStaticProps)
   Avoid heavy SSR where possible
   Reduce bundle size (important in Lambda cold start)

10. Testing

    Local:
	npx open-next dev

    After deploy:
	Test API routes
	SSR pages
	Static assets
	Edge caching

Architecture Overview
	User → CloudFront → Lambda (Next.js SSR)
        	             ↓
                	   S3 (static assets)

./open-next.config.ts:1:30 Type error: Cannot find module 'open-next' or its corresponding type declarations.

This error is not from Next.js itself, but from TypeScript trying to validate open-next.config.ts.

During next build, TypeScript scans all .ts files in the project.

👉 It fails because:
	open-next is not installed, OR
	It doesn’t expose TypeScript types

Fix Options
✅ Option 1 (Recommended): Install OpenNext
	npm install open-next
	npm run build


✅ Option 2: Exclude Config from Type Checking (Best Practice)
	Since open-next.config.ts is not required for Next.js build, exclude it.

	Update tsconfig.json:
	{
  		"exclude": ["node_modules", "open-next.config.ts"]
	}

✅ Option 3: Rename to .js (Simple & Effective)

	Avoid TypeScript validation entirely:

	mv open-next.config.ts open-next.config.js

	Update file:

	module.exports = {
  		aws: {
    			region: "ap-south-1",
  		},
	};

	👉 This is the most common approach in production
